# Tech Stack, Infrastructure & DevOps Specification

**Product:** AgentReady API Layer (MCP Server + Eval Harness)
**Scope:** MVP infrastructure — built for a solo founder, low cost, fast iteration
**Version:** 1.0

---

## 1. Overview

This document defines the concrete technology choices for building, running, and operating the MVP: language/runtime, frameworks, database, hosting, CI/CD, monitoring, and secrets management. Every choice prioritizes **low operational overhead and low cost** over scalability — scaling concerns are noted separately as future work.

## 2. Core Tech Stack

### 2.1 Language & Runtime

| Component | Choice | Notes |
|---|---|---|
| Language | TypeScript | Type safety for tool schemas and API client wrappers |
| Runtime | Node.js 20 LTS | Stable, well-supported by MCP SDK and Fly.io/Railway |
| Package manager | pnpm | Efficient monorepo workspace support |

### 2.2 Core Libraries

| Purpose | Library | Notes |
|---|---|---|
| MCP server framework | `@modelcontextprotocol/sdk` | Official Anthropic SDK; supports SSE/HTTP transport |
| Anthropic API client (for evals) | `@anthropic-ai/sdk` | Used by eval harness to run Claude against MCP servers |
| HTTP client (calling target APIs) | `undici` or native `fetch` (Node 20+) | Lightweight, no extra dependency needed |
| Schema validation | `zod` | Validate tool inputs at runtime; can derive JSON Schema for tool definitions |
| Logging | `pino` | Structured JSON logging, low overhead |
| Retry/backoff | custom wrapper (`withRetry`) in `@agentapi/core` | Exponential backoff for rate-limited target APIs |
| Testing | `vitest` | Unit tests for tool handlers and client wrappers |

### 2.3 Landing Page / Directory

| Component | Choice | Notes |
|---|---|---|
| Framework | Next.js (App Router) | Static generation for directory pages; fast, SEO-friendly for content write-ups |
| Styling | Tailwind CSS | Fast iteration, matches frontend-design conventions |
| Content for eval scores | Static JSON files committed to repo, regenerated by eval harness | No database needed for MVP — scores update via CI |
| Blog/write-ups | MDX files in repo | Simple, version-controlled content |

## 3. Database Requirements

### 3.1 MVP recommendation: **No database required initially**

For the MVP scope (2–3 MCP servers, a static directory, and eval results), a database is not strictly necessary:

- **Eval scores & task results** → stored as JSON files in the repo (`eval-harness/results/*.json`), regenerated on each eval run, consumed statically by the Next.js site at build time.
- **MCP server configuration** (which tools, which target API) → defined in code, not data — no runtime config storage needed.
- **No user accounts, no marketplace** → no need for user data persistence in MVP.

This keeps infrastructure minimal: **zero database servers to provision, secure, or pay for during MVP validation.**

### 3.2 When a database becomes necessary (post-MVP triggers)

| Trigger | What's needed |
|---|---|
| Marketplace with user accounts/subscriptions | Postgres (user accounts, subscriptions, API access tokens) |
| Per-user OAuth tokens for target APIs | Encrypted credential storage (Postgres + encryption at rest, or a secrets manager like Doppler/Vault) |
| Usage analytics / rate limiting per customer | Postgres or a time-series store (e.g., Tinybird, ClickHouse) for request logs |
| Dynamic, self-serve onboarding of new APIs | Postgres to store per-tenant integration configs |

**Recommended choice when needed:** **Neon** or **Supabase** (managed Postgres, generous free tiers, easy to integrate with Next.js and Node services). Avoid self-hosting a database for as long as possible — not worth the operational overhead pre-revenue.

## 4. Hosting & Infrastructure

### 4.1 MCP Servers (one deployment per target API)

| Aspect | Choice | Rationale |
|---|---|---|
| Hosting | **Fly.io** (preferred) or Railway | Cheap, supports long-running Node processes with persistent connections (needed for SSE) |
| Packaging | Docker (one `Dockerfile` per `mcp-server-*` package) | Consistent builds, easy to redeploy independently |
| Transport | SSE over HTTPS, exposed on `:8080` | Required for remote MCP connections (not stdio) |
| Region | Single region (closest to target API's region, e.g., `us-east` if target API is US-hosted) | Minimizes latency to upstream API; MVP doesn't need multi-region |
| Scaling | 1 instance per server, `min_machines_running: 1` | Avoid cold-start delays for agent connections; MVP traffic is low |

### 4.2 Landing Page / Directory

| Aspect | Choice | Rationale |
|---|---|---|
| Hosting | **Vercel** | Native Next.js support, free tier sufficient for MVP traffic, automatic preview deployments |
| Domain | Custom domain (e.g., `agentready.dev`) via Vercel DNS or external registrar | Branding + credibility for outreach |

### 4.3 Eval Harness

| Aspect | Choice | Rationale |
|---|---|---|
| Execution environment | Local machine or GitHub Actions runner | No persistent hosting needed — runs on demand or on schedule |
| Trigger | Manual (`pnpm run eval`) during development; optional scheduled GitHub Action (e.g., weekly) once stable | Keeps cost at zero; scheduled runs catch regressions if target APIs change |
| Output | Writes JSON results to `eval-harness/results/`, committed via PR or direct push | Feeds the static landing page data |

## 5. DevOps Requirements

### 5.1 Repository Structure & CI/CD

- **Monorepo** (pnpm workspaces) containing all `mcp-server-*` packages, `eval-harness`, `landing-page`, and `@agentapi/core`.
- **CI (GitHub Actions):**
  - On every PR: run `lint`, `typecheck`, and `vitest` unit tests for all packages.
  - On merge to `main`:
    - Build & deploy changed `mcp-server-*` packages to Fly.io (using `flyctl deploy`, scoped per package via path-based triggers).
    - Deploy `landing-page` to Vercel (automatic via Vercel's GitHub integration).
  - **Optional scheduled job:** weekly eval run → commits updated scores to `eval-harness/results/` → triggers landing page rebuild.

### 5.2 Environment & Secrets Management

| Secret | Where stored | Used by |
|---|---|---|
| Target API credentials (per integration) | Fly.io secrets (`flyctl secrets set`) | Respective `mcp-server-*` |
| Anthropic API key | GitHub Actions secrets + local `.env` (gitignored) | Eval harness |
| Vercel environment variables | Vercel project settings | Landing page (if any runtime config needed) |

**Rule:** No credentials ever committed to the repo. Use `.env.example` files to document required variables without values.

### 5.3 Monitoring & Logging

| Aspect | Tool / Approach | Notes |
|---|---|---|
| Application logs | `pino` → stdout, captured by Fly.io's built-in log aggregation | Sufficient for MVP; view via `flyctl logs` |
| Uptime monitoring | **Better Uptime** or **UptimeRobot** (free tier) | Simple HTTP health-check pings on each MCP server's `/health` endpoint |
| Error tracking (optional, post-MVP) | Sentry (free tier) | Add once MCP servers have real external traffic beyond evals |
| Eval result tracking | Versioned JSON in repo (git history = audit trail) | No external tool needed for MVP |

### 5.4 Health Checks

Each MCP server should expose a simple `/health` endpoint (separate from the MCP/SSE endpoint) returning `200 OK` with basic status (uptime, target API reachability check). Used by:
- Fly.io's built-in health checks (to restart unhealthy instances)
- External uptime monitoring

## 7. Future Infrastructure Considerations (Post-MVP)

These are **not required for MVP** but are flagged so architecture decisions don't block them later:

- **Database introduction:** Add Neon/Supabase Postgres when user accounts, per-user OAuth tokens, or a marketplace are built. The `@agentapi/core` package should keep an abstraction layer (e.g., a `ConfigProvider` interface) so swapping from "static config" to "DB-backed config" doesn't require rewriting tool handlers.
- **Multi-region deployment:** If MCP servers serve global agent traffic with latency-sensitive workloads, consider Fly.io's multi-region deployment (it supports this natively without major architecture changes).
- **Secrets manager upgrade:** Move from platform-native secrets (Fly.io/Vercel) to a dedicated secrets manager (Doppler, AWS Secrets Manager, or HashiCorp Vault) once managing credentials for many integrations and/or per-user tokens.
- **Observability upgrade:** Add Sentry for error tracking and a proper metrics/dashboard tool (e.g., Grafana Cloud free tier) once there's real third-party traffic to monitor beyond your own eval runs.
- **Containerized eval harness:** If eval suites grow large or need to run frequently, move from local/GitHub Actions execution to a scheduled job on Fly.io or a small dedicated worker.
